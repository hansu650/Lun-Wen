## 3. batch 长什么样

假设：

- `batch_size = 2`

那么 DataLoader 送进模型的大致是：

```python
batch = {
    "rgb":   [2, 3, 480, 640],
    "depth": [2, 1, 480, 640],
    "label": [2, 480, 640],
}
```

可以这样理解：

- `rgb`：彩色图
- `depth`：深度图
- `label`：每个像素的类别编号

在 `BaseLitSeg.training_step()` 里，第一步就是把它拆开：

```python
rgb, depth, label = batch["rgb"], batch["depth"], batch["label"]
```

然后会做：

```python
label = sanitize_labels(label, ...)
```

这一步的作用是：

- 把标签整理到合法类别范围
- 把无效标签设成 `255`
- 让 loss 和指标计算时忽略这些无效像素

你之前已经发现过一个值得长期记住的点：

- 标签里同时出现过 `0` 和 `40`

所以标签边界和 `sanitize_labels()` 的关系，后面如果继续深入，是一个值得复查的地方。

## 4. 从 batch 到 logits

### 4.1 early fusion

如果当前模型是 `early`，核心动作是：

```python
x = torch.cat([rgb, depth], dim=1)
```

也就是：

- `rgb`: `[B, 3, H, W]`
- `depth`: `[B, 1, H, W]`
- 拼接后：`[B, 4, H, W]`

然后统一送进编码器，提多尺度特征，再交给 decoder，最终得到：

```python
logits: [B, 40, H, W]
```

### 4.2 mid fusion

如果当前模型是 `mid_fusion`，路径变成：

```text
rgb -> RGBEncoder -> rgb_feats
depth -> DepthEncoder -> depth_feats
rgb_feats + depth_feats -> GatedFusion -> fused_feats
fused_feats -> decoder -> logits
```

这里的关键区别是：

- `early`：先拼输入，再统一提特征
- `mid_fusion`：先各自提特征，再中间融合

但不管是哪条线，最终都会走到：

```python
logits: [B, 40, H, W]
```

---

## 5. 从 logits 到 loss

有了：

- `logits`: `[B, 40, H, W]`
- `label`: `[B, H, W]`

就可以计算：

```python
loss = CrossEntropyLoss(ignore_index=255)(logits, label)
```

这里的意思是：

- 每个像素都在做一次 40 分类
- `255` 位置不参与损失

所以 `loss` 本质上就是：

> 当前这批图里，所有有效像素平均分错得有多严重。

---

## 6. 从 logits 到 pred，再到 mIoU

验证和评估时，需要把 `logits` 变成真正类别图：

```python
pred = logits.argmax(dim=1)
```

于是：

- `logits` 是每类分数图
- `pred` 是最终类别图

再结合真实 `label`，就可以构造混淆矩阵，并进一步计算：

- 每类 IoU
- 平均 IoU（mIoU）

IoU 的核心公式是：

```text
IoU = 交集 / 并集
```

在语义分割里可以理解成：

- 交集：某类被正确分到的区域
- 并集：真实是这个类或被预测成这个类的区域总和

## 7. 训练、评估、推理三件事分别在干什么

### 7.1 `train.py`

负责把这些零件组装起来：

- 参数解析
- DataModule
- 模型
- checkpoint
- early stopping
- Lightning Trainer

最后调用：

```python
trainer.fit(model, datamodule=datamodule)
```

### 7.2 `eval.py`

作用是：

- 加载指定 checkpoint
- 在测试集跑一遍
- 输出 `mIoU` 和 `PixelAcc`

### 7.3 `infer.py`

作用是：

- 加载指定 checkpoint
- 取若干样本做预测
- 保存可视化图片

这 3 个文件对应的是一条很标准的工程主线：

```text
训练 -> 定量评估 -> 定性可视化
```

## 10. 怎么看 TensorBoard

你现在打开 TensorBoard 后，最值得盯的是这些曲线：

- `train/loss_epoch`
- `val/loss`
- `val/mIoU`

最基本的看法是：

- `train/loss_epoch` 下降，说明模型在学
- `val/mIoU` 上升，说明验证集表现变好
- 如果训练 loss 很低但验证没怎么涨，可能开始过拟合

你现在最适合的使用方式是：

- 把 `early_run1` 和 `mid_run1` 分开看
- 先比较最优 `val/mIoU`
- 再看谁上升更稳、谁更早平台期

## 11. 你已经踩过、以后还会遇到的几个小坑

### 11.1 相对路径坑

- 脚本在哪不重要
- 运行时工作目录和 `data_root` 才重要

### 11.2 Windows 路径分隔符坑

你现在的 checkpoint 文件名里用了 `val/mIoU`，所以在 Windows 下会被拆成子目录。  
这不影响结果，但目录结构会比较怪。

### 11.3 eval 数值和训练日志不完全一致

训练时的 `val/mIoU` 和 `eval.py` 的 `mIoU` 统计口径不一定完全一样，所以数字可能有偏差。  
这个不是一定有 bug，先要分清“统计方式是否一致”。

## 12. 一句话总流程

你后面如果忘了细节，就先回到这条最短主线：

```text
DataModule 读入 RGB / Depth / Label
-> 模型前向得到 logits
-> CrossEntropyLoss 训练
-> argmax 得到 pred
-> pred 和 label 统计 mIoU
-> 用 eval 看数值
-> 用 infer 看预测图
-> 用 TensorBoard 看训练曲线
```
