# 第三次课总结：后续模型改进方向

## 1. 这次课的主要目标

第三次课主要不是总结当前已经跑出来的结果，而是讨论后续模型还能往哪些方向改。

老师给出的核心思路是：现在不能只是简单堆模块，而是要想清楚后面论文的故事怎么讲。后续改进最好围绕一个清楚的主线展开，让模型结构、实验设计和论文叙述能够对应起来。

本次课可以整理出四个主要改进方向：

1. backbone 分工调整：RGB 用 DINO，depth 用 Swin Transformer。
2. depth 输入预处理：尝试 HHA/HHT 这类几何预处理。
3. 融合模块统一化：每个 stage 使用相同融合模块，并在模块内部增加设计。
4. 模型轻量化验证：backbone 全部先换成 Tiny 版本。

## 2. 改进方向一：backbone 重新分工

当前思路可以从原来的结构中跳出来，重新考虑 RGB 和 depth 两个模态分别适合什么 encoder。

新的方向是：

- RGB 分支使用 `DINO`
- depth 分支使用 `Swin Transformer`

这样做的原因是：

- RGB 图像包含纹理、颜色、物体外观和语义信息，比较适合用 DINO 这种强预训练视觉模型提取语义 token。
- depth 图像更强调空间结构和几何关系，可以尝试用 Swin Transformer 这种分层结构来提取多尺度特征。
- 这样可以避免只是把 DINO 和 Swin 机械并列，而是让两个 backbone 有更清楚的模态分工。

论文里可以这样讲：

> RGB branch uses DINO to extract strong semantic token representations, while depth branch uses Swin Transformer to encode hierarchical geometric features.

也就是说，DINO 负责 RGB 语义，Swin 负责 depth 几何结构。

## 3. 改进方向二：depth 预处理

第二个方向是对 depth 输入先做预处理，而不是直接把原始 depth 图送进网络。

课堂笔记里提到了 depth 预处理方向，可以尝试类似 `HHA` 或 `HHT` 的几何编码方式。

这一类预处理的目的不是单纯改变输入格式，而是把 depth 中隐含的几何信息显式表达出来，例如：

- 深度变化
- 空间高度
- 表面方向
- 几何边界
- 局部结构关系

这样做的好处是：

- depth 分支得到的信息更接近几何语义，而不是只有原始深度数值。
- 对 RGB-D 分割来说，几何结构有助于区分物体边界和空间布局。
- 后续融合模块可以更容易利用 depth 信息。

后面实验可以设置两种对比：

- 原始 depth 输入
- depth 经过 HHA/HHT 预处理后的输入

如果预处理有效，就可以把它写成 depth branch 的一个重要改进点。

## 4. 改进方向三：融合模块统一化

第三个方向是融合部分。

老师提到，融合机制本身可以做得更复杂一点，但结构上最好不要太散。也就是说，不要每个 stage 都换一个完全不同的模块，而是考虑：

> 四个 stage 都使用同一种融合模块。

这样做的好处是：

- 结构统一，论文更好写。
- 消融实验更清楚。
- 可以证明提出的融合模块本身有效，而不是靠某一层特殊设计偶然提升。

融合模块内部可以增加更多设计，例如：

- channel attention
- spatial attention
- cross attention
- depth-aware gating
- token-level interaction
- 全局 pooling 或 class token 引导

但整体接口最好保持一致：

```text
RGB feature + depth feature -> fused feature
```

四个 stage 都使用相同逻辑，只是输入通道和空间尺度不同。

论文里可以把这个部分作为主要创新点：

> We design a unified RGB-D fusion block and apply it consistently across all stages to enable multi-level semantic-geometric interaction.

这里需要注意，cross attention 设计时要明确谁是 `Q`，因为 cross attention 的输出结果主要来自 `Q` 所在的特征空间。

## 5. 改进方向四：backbone 全部换成 Tiny

第四个方向是模型规模调整。

课堂笔记里提到“把 B 换成 T”，可以理解为先把 Base 版本 backbone 换成 Tiny 版本。

也就是：

- `DINO-B` 先换成 `DINO-T`
- `Swin-B` 先换成 `Swin-T`

这样做的原因是：

- Base 模型训练成本高，调参和消融都比较慢。
- Tiny 模型更适合快速验证结构是否有效。
- 如果一个融合模块在 Tiny backbone 上都没有提升，就不一定值得放到 Base 上继续跑。
- 如果 Tiny 上有效，再迁移到 Base，论文说服力会更强。

所以后续更合理的实验路线是：

1. 先用 Tiny backbone 快速验证新结构。
2. 确认有效后，再换回 Base backbone。
3. 最后用 Base 版本做正式对比实验。

## 6. 四个方向之间的关系

这四个方向不是互相独立的，而是可以组合成一条新的模型路线：

```text
RGB input
  -> DINO-T / DINO-B
  -> RGB semantic tokens or features

Depth input
  -> HHA/HHT preprocessing
  -> Swin-T / Swin-B
  -> depth geometric features

RGB feature + depth feature
  -> unified fusion block at each stage
  -> decoder
  -> segmentation result
```

这个路线比单纯“换模块”更清楚，因为它有明确分工：

- DINO 提取 RGB 语义。
- depth 预处理增强几何信息。
- Swin 提取 depth 多尺度结构。
- 统一 fusion block 负责 RGB-D 交互。
- Tiny backbone 用来降低实验成本。

## 7. 后续实验建议

后续实验最好按顺序做，不要一次性全部改完。

建议实验顺序：

1. 先把 backbone 全部换成 Tiny，确认训练流程能跑通。
2. 再尝试 RGB-DINO、depth-Swin 的分工调整。
3. 然后加入 depth 预处理，看 HHA/HHT 是否有效。
4. 最后设计统一融合模块，让四个 stage 使用同一个 fusion block。

每次实验只改一个主要变量，这样结果更容易分析。

例如：

- 实验 A：只换 Tiny backbone。
- 实验 B：在 Tiny 上测试 RGB-DINO + depth-Swin。
- 实验 C：在实验 B 基础上加入 depth 预处理。
- 实验 D：在实验 C 基础上替换统一融合模块。

这样后面写论文时，消融表也会比较清楚。

## 8. 总结

第三次课总结出的后续改进方向主要有四个：

- backbone 重新分工：RGB 用 DINO，depth 用 Swin Transformer。
- depth 输入预处理：尝试 HHA/HHT，增强 depth 几何表达。
- 融合模块统一化：四个 stage 使用相同 fusion block，并在模块内部增加 attention、gate 或 token interaction。
- 模型轻量化验证：先把 backbone 从 Base 换成 Tiny，降低实验成本。

这四点可以组成后续模型改进的主线。重点不是盲目把模型做复杂，而是让每个部分都有明确作用：RGB 负责语义，depth 负责几何，fusion 负责交互，Tiny 版本负责快速验证。
